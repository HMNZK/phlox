# 0030: トランスクリプトの非遅延レイアウト化と composer 計測の一方向化（CPU 暴走根治）

> **このファイルの役割**: チャット UI の CPU 暴走（sustained 55-100% 固着ハング）を根治した4つの決定と、その実機証拠・棄却案・診断の訂正を不変記録する。
> **書かないもの**: 現行構造の説明（→ architecture/chat-orchestration.md 等）・作業経緯（→ delivery/0013）。

## 文脈

composer 統一作業（ADR 0029 系）後、「左右サイドバー展開＋グリッド表示＋長文セッションの反復スクロール」で CPU が 55-100% に固着しアプリがハングした。swift test・grep・静的解析では検出不能で、実機 Debug 起動＋`sample`／`top` ポーリングによる単一変数の対照実験（E1〜E7、2026-07-05）でのみ因果を確定できた。

## 決定

### 1. トランスクリプトのスタックは LazyVStack ではなく **VStack**（非遅延）にする

- **真因（実機確定）**: LazyVStack は「更新中のセッションタイルがある状態でのスクロール」を契機に、〔行の実体化/破棄 → 推定高と実測高の差を補正する anchor translation → 可視窓ズレ → 別の行が実体化/破棄…〕という**レイアウト不動点が成立しない自走ループ**に入る。一度入ると契機（実行中セッション）が消えても永遠に continuing（body 再評価ゼロ・state 書込ゼロの純レイアウトループのため `_printChanges` にも映らない）。
- **証拠**: idle＋スクロール=収束0% ／ running 単独=収束0% ／ running＋スクロール（LazyVStack）=発火後 99% 自走 ／ 同条件で VStack=収束 max16.8%・スクロール体感差なし。
- **棄却案**: (a) List 化（NSTableView 系・見た目再現コスト大）、(b) Lazy 維持＋anchor 安定化（安定化 API が不在・v1 で幅制約系4手法全滅の実績）、(c) 放置（ハング容認不可）。
- **トレードオフ**: 全行を先行レイアウトするため極端に長いトランスクリプトでは表示開始コストが増える。問題化した場合は**末尾 N 件のみ描画（件数制限）**で対処し、遅延機構は再導入しない。
  - **更新注記（2026-07-08）**: 監査所見 P1 で問題化が確認され、この件数制限を **ADR 0051**（TranscriptWindow・既定200件・段階展開）として実装した。遅延機構は再導入していない。

### 2. composer / トランスクリプトの幅は**親から演繹した純関数**で決める（計測フィードバック禁止）

- 旧実装は自身の幅を `.background(GeometryReader)` で計測して @State に書き、その値をレイアウトへ戻す往復（計測→書込→再レイアウト）で body 無効化サイクルの起点になっていた（`_printChanges` 実機ログで `_mainColumnWidth changed` が起点と確定）。
- 幅の単一真実源は `ComposerLayout`（純関数）: `maxWidth` = 60%（上限800・狭幅時90%）、`transcriptContentMaxWidth` = その恒等別名（受け入れテストで全域一致を固定）。単一表示・グリッドとも GeometryReader の読み取り値をインラインで渡すのみ。

### 3. AppKit ブリッジ（IMESafeTextView）の高さ書込は**描画パスで行わない**

- `updateNSView`（view update 中）での @Binding 同期書込は SwiftUI に落とされ、送信後に入力欄高さが戻らない Bug A の原因だった。高さ決定は純関数 `ComposerHeightPolicy`（クランプ・ceil・0.5pt 差分ガード）に一本化し、書込はイベント文脈（textDidChange）と「差分がある時だけの遅延 Task（実行時再計算・高々1回で固定点収束）」に限定。

### 4. トランスクリプト行内に**非同期に自身のサイズを変える View を置かない**（防御原則）

- HighlightSwift `CodeText`（非同期ハイライト完了でサイズが変わる leaf）を同期・決定論的な `ChatCodeHighlighter` に置換し、依存を削除。
- **診断の訂正（重要・正直な記録)**: 当初 E5 実験で「CodeText が駆動源」と確定扱いしたが、これは**実験時間帯に実行中セッションが無かったことによる交絡**だった（CodeText 除去後の E6 でも暴走）。真因は決定1。本置換は「行のサイズ揺れが決定1のループの燃料になる」ことへの防御原則として維持する。

## 結果

- 実機検証（2026-07-05）: グリッド／単一／running＋スクロール複合の全再現条件で 60%超連続ゼロ・放置後一桁%収束。Bug A 解消・要件3（出力列幅=入力欄幅）適用後も収束維持。`swift test` 636 tests green。
- 実装: loopflow run4 task-4〜11（経緯は delivery/0013）。
