# ADR 0046: 入力欄デフォルト高の「80px」はパネル全体の見た目高さと解釈し、エディタ・余白・フッターを一括圧縮する

> **このファイルの役割**: ADR 0044 の決定1（入力欄デフォルト高80px統一）の実装が実機でデグレと判定され、ユーザー確認のうえ覆した決定の記録。ADR 0044 の該当部分を supersede する（実行中インジケータ撤去の決定は不変）。
> **書かないもの**: 高さ決定の現行機構（→ `ComposerHeightPolicy.swift` / ADR 0036・0030）。

## 文脈

ui-brushup-2 run の item4「入力欄が開いた直後に約80pxの高さ」は、`ComposerHeightBounds` の**エディタ最小高**を 44/40 → 80 に変更する形で実装された（`76633ba`・ADR 0044）。同 run の Phase4 実機検証は未実施のままマージされ、teamview-cpu-fix run（2026-07-07）の実機検証中にユーザーが「入力欄の縦幅が広いまま固定されている」デグレとして報告した。

- 実測: エディタ80px＋フッター行＋余白で**パネル全体が約140px**になる。再起動直後・別セッションでも同じ（リサイズ起因の状態バグではなく静的な寸法）。
- ユーザー確認: 「約80px」の意図は**入力欄パネル全体の見た目高さ**（＝以前のコンパクトさ）であり、エディタ単体80pxではない。

## 決定

1. 「80px」要件は「**パネル全体（余白・フッター込み）で約80px**」と解釈し、エディタ単体の最小高で表現しない。
2. パネルを一括圧縮して≈84pxにする: 縦余白 12→**8** ×2・エディタ min **36**（1行ぶん・single/grid 統一）・エディタ↔フッター間隔 8→**4**・フッター高 32→**28**（standard チップ・送信ボタン。compact=24 は不変）。max=160 の auto-grow は維持。
   - 旧コンパクト値（44/40）への単純復帰はパネル≈104〜108でユーザー実測「まだ約120px」と乖離が残ったため不採用（実機で段階確認済み）。
3. 関連する受け入れテスト（`GridComposerAutoGrowAcceptanceTests`・`ComposerHeightPolicyTests`）の期待値を本決定に合わせて更新する（テスト弱体化ではなく、ユーザー決定に基づく契約変更）。

## 棄却案

- **エディタ min=80 の維持（ADR 0044 実装）**: パネル≈140pxでデグレ判定（ユーザー実機確認）。
- **旧値 44/40 への単純復帰のみ**: パネル≈104〜108で「約80px」に届かない（実機で確認し棄却）。
- **エディタ min を中間値に調整のみ**: chrome（余白・フッター32）が支配的で目標に届かない。

## 結果

- 単一/グリッドとも入力欄パネルの安静時高さが≈84px（8+36+4+28+8）になる。auto-grow（〜160）は不変。
- 教訓: 実機検証（Phase4）を silent skip したままマージすると、要件の解釈誤りが検出されない。本 run で3件目の顕在化（P1 表フリーズ・P3 Lazy ループ・本件）。
