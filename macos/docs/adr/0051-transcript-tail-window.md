# 0051: トランスクリプトの末尾 N 件描画（TranscriptWindow・reveal-on-jump・展開アンカー保持）

> **このファイルの役割**: 長大トランスクリプトの描画コストを件数制限で抑える決定と、その3つの派生設計（窓の純関数化・隠れ域ジャンプの可視化・展開位置のアンカー保持）の rationale。
> **書かないもの**: 現行の実装仕様（→ `architecture/chat-mode-ux-components.md`）。

- **文脈**: ADR 0030 は LazyVStack の自走レイアウトループを根治するため非 Lazy VStack（全行先行レイアウト）を採択し、トレードオフとして「極端に長いトランスクリプトでは表示開始コストが増える。問題化した場合は末尾 N 件のみ描画で対処」と自ら規定していた（0030:22）。監査所見 P1（ストリーミング中のデルタ毎に全セル再評価）で問題化が確認されたため、この規定対処を実装した。
- **決定**:
  1. **`TranscriptWindow`（純粋値型）による件数制限**。`visibleRange(totalCount:)` は totalCount のみから末尾スライスの開始位置と隠れ件数を返す純関数（既定 limit=50・expandStep=50。**2026-07-18 改定・ADR 0097**: 初回描画コスト削減とユーザー指摘「200件は多い」を受け 200→50 へ引き下げ・iOS ADR 0022 と同値）。スクロール量・可視領域・GeometryReader には一切連動しない（ADR 0030 の再入禁止を構造で担保）。超過時は先頭に「以前のメッセージを表示（残り k 件）」ボタンを出し、**拡張契機はボタン押下のみ**。セッション切替（vm identity 変化）で既定へ reset。
  2. **reveal-on-jump**: 実行中バックグラウンドタスクへのジャンプ先が隠れ域にある場合、scrollTo の前に `reveal(index:totalCount:)` で window を「必要量＋expandStep のマージン」まで広げる（ぴったり露出だと遅延 scrollTo までの 1 デルタで再び隠れる）。遅延 scrollTo は世代トークン（jumpGeneration）で後続操作・セッション切替時に無効化。
  3. **展開アンカー保持**: 展開ボタン押下時、押下前の先頭可視 item をアンカーとして捕捉し、展開後に `scrollTo(anchor, .top)` をイベント駆動で1回だけ発行する（runtime 確認でユーザーが「展開すると履歴先頭へ飛ぶ」と指摘した対処）。
- **棄却案**: (a) LazyVStack 再導入（ADR 0030 で実機確定済みの自走ループ）、(b) スクロール位置連動の動的 window（レイアウト観測フィードバック＝0030 の再入口）、(c) 隠れ域ジャンプのボタン無効化（対症療法・機能損失）、(d) reveal のぴったり露出（ストリーミング成長で破綻・stage2 レビューが再現テスト付きで棄却）。
- **証拠**: 受け入れテスト PM3Task8TranscriptWindowAcceptanceTests（8件）＋白箱9件 green。実機（Debug 起動・350件 transcript）で表示時 CPU 0.0% 収束・実ストリーミングターンでバースト後即収束・ボタン押下→展開→アンカー保持をユーザー確認済み（2026-07-08）。
- **トレードオフ**: 隠れ域の item は Cmd+F 的な視覚探索から外れる（ボタンで段階的に露出）。expand は単調で縮まない（セッション切替まで）。
