# ADR 0060: 送信済みチャットメッセージの画像添付は「メタのみ永続・バッジ表示」とする

> **このファイルの役割**: チャット履歴のユーザーメッセージに画像添付を可視化する際、実サムネイル（画像バイト永続化）ではなくメタデータ＋バッジを選んだ決定理由と、ChatItem スキーマ拡張の互換戦略。
> **書かないもの**: バッジの描画実装（→ Packages/SessionFeature/Sources/SessionFeature/ChatMessageCells+Basic.swift）。

## 文脈

画像を添付して送信しても、チャット履歴のユーザーメッセージに添付の痕跡が残らなかった。`ChatItem.userMessage` が `(id, text, timestamp)` しか持たず、画像バイトは CLI 送信用 `ChatInput.image` に変換された後 `attachmentStore.clear()` で消える。画像のみ送信に限り「（画像 N 枚）」という代替文字列を本文へ埋めていたが、テキスト＋画像の同時送信では痕跡ゼロだった。

## 決定

- `ChatItem.userMessage` に第4連想値 `attachments: [ChatUserAttachment]`（filename?/mediaType の**メタのみ**）を追加する。**画像バイトは永続化しない**（トランスクリプト JSON の肥大と保存・削除管理の複雑さを避ける）。
- UI はバブル内の軽量バッジ（photo アイコン＋先頭ファイル名（無ければ「画像」）＋2枚以上は ×N）。画像のみ送信の「（画像 N 枚）」プレースホルダは廃止し、テキストは空のままバッジで可視化する（空テキスト時はテキスト行・コピー操作を出さない）。
- スキーマ互換戦略: Swift の enum 連想値にデフォルト値は書けないため、**3引数の静的ファクトリ `userMessage(id:text:timestamp:)` を併設**して既存呼び出しをソース互換に保つ。手書き Codable は「decode 時に欠落→空配列・encode は非空のみ書く」で、**旧トランスクリプト JSON の後方互換と、添付なしメッセージの JSON 形式不変**を両立する。`==` は添付メタを含む。

## 棄却案

- **実画像サムネイル（画像を別ファイル保存）**: 視認性は最高だがストレージ増・保存/削除のライフサイクル管理・履歴復元の I/O が必要で、要求（添付したことが分かる）に対して過剰。ユーザーがゲート①でバッジ案を選択。
- **セッション中のみメモリ上でサムネイル・履歴はバッジ**: 挙動が2形態になり複雑。棄却。
- **本文プレースホルダの拡張**（「（画像 N 枚）」をテキスト＋画像でも付ける）: 表示とデータが未分離のままで、コピー時に混入する。棄却。

## 結果

- 送信後・履歴復元後・再起動後もユーザーメッセージに添付が可視化される（テキスト＋画像／画像のみの両方）。
- 過去の履歴 JSON はそのまま読める（受け入れテスト ChatFixTask7AttachmentBadgeAcceptanceTests が凍結）。
- 新形式の JSON を旧バージョンのアプリが読む前方互換は保証しない（attachments キーは未知フィールドとして無視される設計だが、検証は後方互換のみ）。
