# ADR 0080: アゴラ討論を「完全自由発言（idle ゲート付きリレー）」で実装する

> **このファイルの役割**: n:n エージェント討論の発言制御方式として、ファシリテーター指名制・挙手制を退け「完全自由発言」を選んだ決定・機械的ガードの設計・v1 で受容したトレードオフ。
> **書かないもの**: 現行のコンポーネント構造・型（→ `architecture/team-timeline-view.md`）、エンジンのイベント/コマンド仕様（→ コード内 doc と `AcceptanceAgoraEngineTests`）。

## 文脈

アゴラ（ADR 0072 のフラットグループチャット）に「議題を投入すると複数の Claude Code セッションが討論する」機能を載せる。現実の議論のように「喋りたい者が会話の継ぎ目に割り込む」体験を目指し、ユーザーは3方式（ファシリテーター指名制／挙手制／完全自由発言）を比較検討のうえ「まず理想である完全自由発言でどこまで実装できるか試す」を選択した（実験的 v1）。参加エージェントは v1 では Claude Code（appServer チャット）のみ。データモデルは [agmsg](https://github.com/fujibee/agmsg) を参考にした（「フロアは無知・プロトコルはプロンプトに宿る」、単調増加 seq の追記専用ログ＋参加者ごとの read cursor、ターンの継ぎ目での配送）。

## 決定

**発言制御は「エンジンによる idle ゲート付きリレー」とし、発言の意思決定はエージェント自身（プロンプト規約）に委ねる。**

- **純粋状態機械 `AgoraDiscussionEngine`**（I/O なし・時刻はイベント引数）が唯一の判断者。追記専用の討論ログ（単調 seq）と参加者ごとの cursor を持ち、「生成中・配送待ちの参加者へは配送しない」（idle ゲート）を不変条件とする。
- **配線層 `AgoraDiscussionCoordinator`** は 350ms tick で参加セッションの turn 完了・idle を観測してイベント化し、エンジンが返すコマンド（deliver/summon/requestConclusion/end）を直列実行する。副作用は全て `Effects`（send/injectPrompt/summon）注入で、テストは実セッション不要。
- **発言規約はプロンプト注入**: 発言不要なら本文 `PASS`（ログに載せない・数えない）。役割・議題・上限は spawn 時の役割プロンプトで与える。ファシリテーターは必須参加で、`$PHLOX_CLI spawn --role` による招集・収束・最終まとめを役割プロンプトで担う。
- **機械的ガード**（プロンプト遵守に依存しない側）: 発言数上限 maxUtterances（到達で concluding→最終まとめ→ended）・参加数上限 maxAgents・連続発言制限（他エージェント発言でリセット。ユーザー発言ではリセットしない）・停滞検知（全員 PASS が続いたらファシリテーターへ促し。促しは連続発言制限より優先）・ターンタイムアウト。
- **表示範囲の2モード**: 討論中は参加者集合のみ（階層無視・集合が正）、非討論時は従来（ルート＋直接の子）を正式化。

## 統合検証（実機）で確定した補強

ユニット・受け入れテスト green でも実討論で露見した統合欠陥3件を、実機検証（課金 spawn・ユーザー承認済み）のサイクルで発見・修正した:

- **招集の着地を witness で討論へ自動登録する**: `spawn --role` の成功後、`ControlActionHandler` が `agoraParticipantLanded(id:role:requester:)` を通知し、討論 coordinator へ登録＋参加プロンプト（役割・議題・規約）を注入する。役割なし spawn は「要求者が討論参加者」の場合のみ登録（討論外 spawn の誤登録防止）。これ以前は招集されたセッションが討論に登録されずリレーが死んでいた。
- **role 付き claudeCode spawn の backend 既定は appServer**: 討論リレーは appServer transcript の観測が前提で、PTY セッションは観測できない。明示の `--backend` 指定は引き続き優先。
- **ファシリテーターへの招集コマンドはテンプレートに完全形を記す**（`--kind claudeCode` 固定）。教訓: 実行検証していないコマンド例をプロンプトテンプレートへ書かない（「--kind 省略可」と誤記した結果、実機でファシリテーターの招集が CLI エラーになり Agent ツールでの自己シミュレーションに逸走した）。

## 棄却した代替案

- **ファシリテーター指名制／挙手制**: 実装は安全だが「会話の継ぎ目の割り込み」という理想に届かない。自由発言の実験を v1 の主目標とするユーザー決定により見送り（将来 `AgoraSchedulerKind` の差し替えで導入可能。roundRobin は完了駆動のフォールバックとして実装済み）。
- **常時ブロードキャスト（idle ゲートなし）**: 生成中のセッションへの送信は入力欄破壊・文脈汚染を起こすため不可。
- **エンジンへの I/O 内蔵**: テスト不能・再現不能になるため、純粋状態機械＋Effects 注入に分離。

## v1 で受容したトレードオフ（レビュー裁定済み）

- **send 失敗は best-effort**: 配送失敗時のリトライ・cursor ロールバックはしない（エンジンは配送コマンド発行時点で cursor を前進させる）。失敗は配線層で warning ログに記録し無言にはしない。
- **transcript 置換（revert 等）時**: 境界 itemID 消失を検知したら抽出せず境界を末尾へリセット（過去全発言の再配送を防ぐ安全側）。当該 turn の発言は失われる。
- **同一ポーリング窓に複数 turn**: 1発言に併合されうる（現行の駆動モデルでは実質到達不能と評価）。
- 状態は in-memory のみ（討論の永続化・復元は将来課題）。

## 結果

- 全6タスク（エンジン・抽出・役割注入/role 永続化・coordinator・討論 UI・設定）を per-worktree 受け入れテスト凍結で実装。DashboardFeature 全量 1263 テスト・E2E 15 green。
- 実機検証（課金 spawn・ユーザー承認済み・2026-07-11）: 議題投入→ファシリテーター spawn→役割付き招集（UXデザイナー／パワーユーザー）→着地自動登録→自由発言リレー（`[from 名前]` 帰属・相互応答）→発言カウント→上限到達→concluding→ファシリテーターの最終まとめ→終了、の全フローが実討論で成立することを確認した。発言内容も役割に沿っており自由発言の実験は成立（v1 目標達成）。上の統合欠陥3件はこの検証サイクルで発見・修正したもの。
