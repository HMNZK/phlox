multi-agent orchestration, dissected

AI エージェント同士が
互いを操る仕組み

Phlox は、複数の AI コーディング CLI（Claude Code・Codex・Cursor…）を 擬似端末（PTY）セッションとして走らせる macOS ネイティブアプリだ。 その真価は、あるセッションが $PHLOX_CLI を通じて 別のセッションを spawn し、指示を送り、完了を待ち、結果を統合できること—— すなわち エージェントによるエージェントのオーケストレーションにある。 本書はその内部を解剖する。

6 種の CLI を吸収 2 つのローカル HTTP サーバー posix_spawn + PTY 状態機械 6 状態
00

全体像 — 二つのループで CLI と対話する

子セッションで走る CLI は、Mac アプリ本体と 二つの独立した経路で会話する。 一つは自分から要求を出す制御ループ（$PHLOX_CLI → ControlServer）、 もう一つは自分の状態を通知するフックループ（CLI の hooks → HookServer）。 この二つが揃って初めて「送って、待って、完了を知る」が成立する。

制御ループ

要求を出す

子が $PHLOX_CLI spawn/send/wait/read を叩くと、127.0.0.1 の ControlServer に HTTP + Bearer トークンで届く。各セッションは自分専用のトークンを持ち、それが「誰の要求か」を認可に使われる。

フックループ

状態を通知する

CLI がターン開始・ツール実行・停止といった節目で hooks を撃つと、hook-dispatcher.sh が正規化して HookServer に POST する。これがセッションの running / idle を確定させる。

プロセス／PTY

実体を動かす

送信の実体は擬似端末への書き込み、出力の実体は PTY から流れるバイト列。PTYManager が posix_spawn で子プロセスを起こし、その入出力を仲介する。

01

起動 — テーブル駆動で 6 種の CLI を吸収する

Claude・Codex・Cursor…はコマンド名も引数も完了検知の仕方も違う。Phlox はその差を 一枚の宣言的レジストリに書き下し、起動のコア経路は「どの CLI か」で分岐しない。 差分はデータ、経路は共通——これが多 CLI を破綻なく束ねる要になっている。

差分はレジストリに、経路は共通に

各 CLI 一件分の設定（起動バイナリ・権限バイパス引数・resume の渡し方・hook 方式・idle 判定方式）は AgentRegistry.allDescriptors にハードコードされた唯一のソースにまとまる。 起動時、AgentLaunchPlanner がこのテーブルを読んで argv と環境変数を組み立てるだけだ。

kindバイナリ権限バイパスresume完了検知
claudeCodeclaude—（settings 制御）--resumehook（viaHook）
codexcodex--dangerously-bypass-…先頭 resume <id>出力静止

起動パイプライン — CLI からプロセスまで

phlox spawn --kind X でも UI のボタンでも、経路は途中で spawnNewSession に一本化される。 最終的に posix_spawn が擬似端末付きで子プロセスを起こす。

なぜ posix_spawn か

posix_spawn_file_actions_adddup2 で擬似端末の slave を stdin/stdout/stderr に接続し、addchdir_np で作業ディレクトリを子プロセス側で変更する。親で chdir すると macOS が余計な権限ダイアログを出しうるため、それを避ける設計になっている。Posix.swift:34–104

02

会話 — エージェントがエージェントに指示する

オーケストレーションの手順は Claude の /phlox-cli スキルにまとまっている （以前は全セッションへ毎ターン自動注入していたが、ADR 0035 で廃止し、必要なときに明示参照する方式にした）。 エージェントは次の 6 つの動詞で仲間を操る。

動詞HTTPパス役割
spawnPOST/sessions子エージェントを起動する
sendPOST/send子の端末へテキストを送る（＝指示）
wait-readyGET/sessions/{id}/ready起動完了を待つ（省くと取りこぼす）
waitGET/sessions/{id}/waitターン完了を待つ
readGET/sessions/{id}/output相手の画面テキストを読む
list · rename · killGET/PATCH/DELETE/sessions…一覧・改名・終了

委譲のループ

ガイドが指示する定石はこうだ。spawn → wait-ready → send → wait → read を回し、複数を並行させるときは 先に全員へ spawn・send してから、各 wait をまとめて回収する。

send は「ただの文字列」ではない

send が擬似端末に文字を書き込むまでに、いくつもの関門を通る。無害な送信に見えて、 レート制限・制御文字拒否・送信元の明示・確実な確定（submit）まで面倒を見ている。

安全弁

制御文字は拒否する

本文に \n \r ESC などが含まれると rejected("control-characters") で弾く。委譲メッセージが「必ず 1 行」なのはこのため。MessagingService.swift:117

出所の明示

[from X] を前置する

本文の先頭に送信元名を付ける。ガイドは「先頭の [from X] にそのまま投げ返すな」と釘を刺す——ループ暴走の芽を摘む。

03

完了検知 — 「終わった」をどう知るか

オーケストレーションの心臓は wait だ。だが「相手のターンが終わった」を機械的に知るのは難しい。 Phlox は状態を 6 つの状態と純粋関数の遷移に落とし込み、CLI の流儀に応じて二通りの方法で idle を確定させる。

状態機械 — reduce(status, event)

状態遷移は副作用のない純粋関数 reduce(_:applying:) だ。現在の状態とフックイベントから次の状態が決まる。 これがテスト可能で予測可能な「完了」の定義を与える。

wait が完了とみなす瞬間

running → idle の遷移で completedTurnSeq が 1 つ進む。wait は開始時のシーケンス値を基準に 100ms ごとにポーリングし、この値が基準を超え、かつ承認待ちでなければ done を返す。イベント購読ではなくポーリングで、hook 方式・出力静止方式のどちらでも同じ「turn++」に集約される。DashboardViewModel.swift:1346–1407

idle を確定させる二つの方法

Claude は hooks で明示的に「停止した」と教えてくれる。だが Codex・Cursor はそれを撃たない。 そこで Phlox は出力が一定時間静止したら idleとみなす方式を併走させる。同じ turn++ に着地するのがミソだ。

Claude · viaHook

フックで確定する

  • SessionStart が来るまで starting から抜けない（フォールバックなし）
  • Stop フックで running → idle、turn 完了
  • ESC 単体入力は Stop を出さないため、lone-ESC で idle 復帰する特例補正あり
Codex · Cursor · idleOnSpawnComplete

出力静止で確定する

  • spawn 完了で即 idle。入力可否は「出力が落ち着いたか」だけで判定
  • 送信後、出力が届くたびに 400ms のタイマーを張り直す（世代カウンタで最新だけ有効）
  • 最後の出力から 400ms 静止で running → idle、turn 完了

支配する時間定数

この orchestration の「間合い」は、いくつかの小さな定数で決まっている。

定数値役割
出力静止 settle 窓400 msCodex/Cursor の idle 確定（最後の出力からの静止時間）
submit キー遅延250 msbracketed paste 後、CR を送るまでの待ち（Enter 抑制窓を跨ぐ）
done ポーリング間隔100 mswait が turn 完了を検出する周期
ready ポーリング間隔20 mswait-ready が入力可を検出する周期

04

暴走防止 — 木・深さ・認可

エージェントがエージェントを起動できるということは、放置すれば指数的に増殖しうるということでもある。 Phlox は親子の木構造・上限・認可の三段で、この危うさを機械的に抑え込む。

増殖を抑える

三つの上限

総数 16・深さ 3・毎秒 5。spawn 時に レート記録 → 総数/深さ検証の順で判定し、超過は 429（レート/総数）・403（深さ）で拒否する。

誰が誰を殺せるか

祖先だけが kill できる

kill は、要求元が対象の祖先である場合のみ許可。無関係なセッションを終了させられない。親リンクを visited 付きで辿って判定する。SpawnPolicy.swift:70–97

循環に強い

木の構築も安全

森の構築は activePath で循環検出、emittedIDs で重複排出を防ぐ。親が見つからない孤児も root として救済する。SessionTree.swift:64–153

認可の土台 — セッション単位トークン

spawn 時に 32 バイト乱数のトークンを発行し、子プロセスの env PHLOX_TOKEN として渡す。ControlServer は Authorization: Bearer を必須検証し、トークンから「要求元セッション」を解決する。これが深さ計算・kill 認可・自己送信判定・レート制限キーのすべての基礎になる。SessionTokenStore · ControlServer.swift:151–165

用語

制御プレーン — 子が要求を出す経路。$PHLOX_CLI → ControlServer。spawn/send/wait/read。

フックプレーン — 子が状態を通知する経路。CLI hooks → HookServer。running/idle を確定。

turn（ターン） — 1 回の対話往復。completedTurnSeq が完了で 1 増え、wait の合図になる。

bracketed paste — 端末の貼り付けモード（CSI 200~/201~）。長文送信の末尾 Enter 取りこぼしを防ぐ。
